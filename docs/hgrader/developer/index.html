Developer Guide

HGrader Architecture

Codebase structure, data models, services, and build configuration for the fully open-source HGrader iOS app.

Tech Stack

LayerTechnology
UISwiftUI with NavigationStack for hierarchical navigation and deep linking between session list, detail, and settings.
PersistenceSwiftData with ModelContainer; schema entities GradingSession, QuestionRubric, and StudentSubmission.
PlatformiOS 17.0+, Swift 5.0
IDEXcode 15.4+
OpenAIResponses API and Batch API over URLSession (HTTPS JSON). Live calls use structured outputs; batch jobs use JSONL file upload and polling.
CaptureVisionKit document scanner for sheet capture; custom AVCaptureSession-based camera path where the scanner is not used.
SecretsOpenAI API key in Keychain via the Security framework (SecItemCopyMatching, add/update/delete); see Security.
BackgroundBackground App Refresh for pending batch job refresh; local notifications when batches complete (see Background Tasks).

Data Models

HGrader persists three SwiftData @Model types. Cascade deletes ensure rubric rows and submissions are removed when a session is deleted. Large binary payloads use external storage where noted so the primary store stays lean.

OpenAI Integration

Networking is centralized in OpenAIService and OpenAIService+Batching.swift, using URLSession against the OpenAI REST API.

Endpoints

PathUse
/v1/responsesLive structured completions (rubric, grade, validation) with JSON schema output.
/v1/batchesCreate batch jobs and poll status until terminal state.
/v1/filesUpload JSONL inputs (purpose: batch) and download result files by output_file_id.
/v1/organization/costsOrganization usage/cost fetch for Settings (when the key has access).

Live API flow

Images are encoded as base64 and sent in the message content alongside system instructions and user text. Requests target /v1/responses with structured output definitions so the model returns JSON matching app-defined schemas (rubric shape, per-question grades, validation verdicts).

Batch API flow (recommended path)

  1. Build JSONL — Write a temporary file; each line is one batch request object with a custom_id, method: POST, url: /v1/responses, and a body that references student images via file URLs (after upload) rather than inlined base64 for scale.
  2. Upload — POST /v1/files with purpose: batch to obtain input_file_id.
  3. Create batch — POST /v1/batches with input_file_id, endpoint /v1/responses, and completion_window: 24h (or as configured).
  4. Poll — Poll batch status until completed or failed.
  5. Download — Fetch output JSONL from /v1/files/{output_file_id}/content.
  6. Parse — For each JSONL line, read the HTTP-style response body and extract the model output text / structured payload for merging into SwiftData.

Batch cost optimization

Batch API pricing is lower than synchronous Responses calls. The app applies a 0.5 multiplier to usage estimates for batch lines when projecting cost, so UI estimates stay aligned with the discounted batch path.

Model pricing

PricingCatalog in OpenAIServiceTypes.swift defines ModelPricing (per-million-token USD rates for input, cached input, and output). It matches model IDs by prefix, including GPT-5.4 / 5.2 / 5 (plus mini and nano variants), GPT-4.1 (and mini), and GPT-4o (and mini). estimatedCostUSD(modelID:inputTokens:outputTokens:cachedInputTokens:) combines usage with these rates.

Validation pipeline

After initial grading, if validation is enabled, a separate validation model reviews scores and reasoning. If validation rejects or requests changes, the app can regrade using the prior grading output plus validator feedback. This loop runs up to validationMaxAttempts before surfacing failure or manual review in the UI.

API key management
The key lives only in Keychain under AppSecrets.openAIKey (see Security). It is never written to SwiftData, export packages, or debug archives intended for sharing.

Grading Pipeline

Core orchestration lives in SubmissionProcessing.swift.

  • SubmissionProcessor.grade() — Entry point: runs initial gradeSubmission, then delegates to processValidatedSubmission when validation is enabled.
  • Validation path — processValidatedSubmission runs validate → approve or enter a regrade loop (prior grade + validator feedback) up to validationMaxAttempts.
  • SubmissionDebugRecorder — An actor that records a chronological trace of pipeline events for debugging; output can be persisted to submission external storage when enabled.
  • BatchSubmissionStage — Enum modeling async stages: queued → grading → validating → regrading → terminal completed or failed. Persisted via batchStageRaw on StudentSubmission.

Batch Processing

Session-level batching is split across SessionDetailView+Batching.swift and OpenAIService+Batching.swift.

Rubric batch

Async rubric generation uses session fields rubricProcessingStateRaw, rubricRemoteBatchID, and rubricRemoteBatchRequestID, plus external archives for pending input. Helpers such as refreshPendingRubricGeneration poll OpenAI until the job finishes; finalizeCompletedRubricBatch downloads results, parses JSONL lines, and materializes QuestionRubric rows.

Submission batches by stage

Submissions move through stages: grading, then validation, then regrading when needed. Each stage can enqueue a new OpenAI batch. On completion, the UI logic auto-queues the next stage so teachers see continuous progress without manual restarts.

UI polling hooks

SessionDetailView keeps identifiers such as pendingRubricPollingID and pendingBatchPollingID so .refreshable and foreground refresh can re-query batch status without duplicating in-flight work.

View Architecture

ContentView

  • Data — @Query over GradingSession, sorted by createdAt descending.
  • Navigation — NavigationStack with destinations for session detail.
  • NewSessionSheet — Creates sessions with model selection from ModelCatalog and API-related settings aligned with the session model fields.
  • SettingsView — Keychain-backed API key editor, organization cost fetch (/v1/organization/costs), and tracked in-app cost keyed by APIKeyIdentity.fingerprint(for:) (see Security).

SessionDetailView

  • Tabs — Segmented control bound to SessionSectionTab (.rubric, .overview, .results) with a TabView (page style) for swiping between sections.
  • Forms — Three primary forms; each uses .refreshable to poll batch and job status when the user pulls to refresh.
  • ScanIntent — Enum .master, .student, .batch routes capture flows to the correct archive fields and follow-up processing.

Supporting views include SubmissionReviewViews for per-submission summaries, SavedSubmissionDetailView for deep inspection and score edits, AnswerKeyReviewView and RubricEditors for rubric approval and edits, and DocumentScanner as the VisionKit bridge.

Export Format

SessionExporter (in SessionExporters.swift) assembles a directory tree, then zips it.

  • session.csv — Flat spreadsheet of students and scores for quick review.
  • rubric.json — Machine-readable rubric definition.
  • session-summary.json — Session metadata and aggregates for downstream tools.
  • master_scans/ — Exported master answer-key images.
  • submissions/<safeName>-<idPrefix>/ — Per student: summary.json plus scans/ for page images.

Archiving uses SimpleZipWriter: a minimal ZIP implementation emitting local file headers, a central directory, and end-of-central-directory record, with per-entry CRC32 for integrity. This avoids third-party zip dependencies while staying compatible with standard unzip tools and CanvasConnect ingestion.

Security

  • Keychain account — AppSecrets.openAIKey (AppSettings.swift) is the keychain account string for the OpenAI credential.
  • KeychainStore — Singleton in AppSettings.swift wrapping SecItemCopyMatching, SecItemUpdate, SecItemAdd, and SecItemDelete for read/write/delete of the API key.
  • Identity without exposure — APIKeyIdentity.fingerprint(for:) (CryptoKit SHA256) returns a 16-character hex prefix of the digest for correlating usage and settings display, or nil for empty keys. The full key never appears in logs, SwiftData, or exports.
  • Exports — Session ZIP and JSON artifacts omit secrets; only grades, rubric, and image payloads leave the device.

Background Tasks

  • AppBatchRefreshCoordinator — Schedules BGAppRefreshTaskRequest so iOS can wake the app periodically. When the task runs, the coordinator refreshes all pending batch jobs (rubric and submission batches) to advance state and persist results.
  • AppNotificationCoordinator — Posts local push notifications when batches reach a completed (or otherwise terminal) state worth surfacing to the teacher, so they need not keep the app foregrounded.

Registration and handling of these tasks is wired from HomeworkGraderApp.swift alongside SwiftData container setup.

Build & Setup

  1. Clone the repository and open HomeworkGrader.xcodeproj in Xcode 15.4+.
  2. Select a physical iPhone as the build target (camera scanning requires a real device).
  3. Build and run. On first launch, configure the OpenAI API key in Settings.
Simulator limitations
The document scanner and camera capture require real hardware. The app builds for the simulator but capture-related features will not function there.

Extension Points

  • Additional AI providers — Introduce a protocol implemented by OpenAIService (or an adapter) to swap base URLs, auth, and response parsing.
  • Custom rubric types — Extend GradingDomain DTOs and JSON schemas for alternate assessment shapes (MCQ, free response blocks, etc.).
  • Direct LMS integration — Canvas is supported via CanvasConnect; the same export package could be uploaded by in-app Canvas APIs if desired.
  • More export targets — Add writers beside SessionExporter / SimpleZipWriter for other gradebook formats.
